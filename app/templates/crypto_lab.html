How to read this dashboard

Every time a doctor sends a chat, saves a patient record, or starts a video call, the system runs the full six-step ECC–Kyber hybrid key fusion pipeline and logs the result here. Different functions use different α/β weightings — a video call prioritises speed (α=0.7), a patient record prioritises quantum resistance (β=0.8), and chat uses the paper's recommended optimal balance (α=0.4, β=0.6). The chart and Table I below are the live version of the paper's evaluation results, built from actual system operations rather than a standalone script.

Interpretation

SPEED_PROFILE is expected to be the fastest of the three because its α=0.7 weighting leans on ECC, the quicker of the two primitives. SECURITY_PROFILE is expected to be the slowest because its β=0.8 weighting leans on Kyber instead. The weighted fusion approach is designed to match or beat a single primitive used alone for its intended purpose: for example, SPEED_PROFILE still keeps some Kyber contribution that a pure-ECC system would not have, without giving up the speed that makes it suitable for a live video call. At the scale these operations actually run at, all configurations land in the low single-digit millisecond range, so individual runs can vary because of normal system scheduling noise rather than a real difference between profiles. This is exactly why the benchmark above runs 20 iterations and reports mean, min, max, and standard deviation instead of a single timed run.

② Replay Attack Resistance

Two operations for the same patient using identical α/β produce different Kfinal values — because SID and timestamp always differ.

What makes each key unique:

SID = uuid4() — random every session

T = datetime.now() — exact time

PID = patient ID — same across sessions

Kfinal = HKDF(Kf ‖ SID ‖ T ‖ PID)

∴ even if Kf repeats, Kfinal never repeats
